# Тесты — разбор для понимания

Этот документ объясняет каждый тестовый файл в папке `tests/`: какой тикет он
покрывает, что происходит в `beforeEach` и что проверяет каждый тест.

> Файлы, начинающиеся с `all_...`, — это старые учебные наработки, они здесь
> не рассматриваются.

## Общие понятия Playwright Test

Прежде чем читать разбор — минимум терминов, которые встречаются в каждом файле.

- **`test.describe('...', () => { ... })`** — группа тестов с общим названием.
  В отчёте все тесты внутри будут сгруппированы под этим заголовком.
- **`test('название', async () => { ... })`** — один тест-кейс.
- **`test.beforeEach(async ({ page }) => { ... })`** — код, который выполняется
  **перед каждым** тестом в группе. Сюда выносят подготовку: логин, открытие
  нужной страницы. Так тесты не повторяют одно и то же.
- **`{ page }`** — «фикстура»: свежая чистая вкладка браузера, которую Playwright
  создаёт для каждого теста заново (тесты не влияют друг на друга).
- **`expect(...)`** — проверка (ассерт). Если условие не выполнилось — тест падает.
- **`await`** — почти все действия асинхронные, поэтому перед ними `await`
  (ждём завершения). Playwright сам ждёт появления элементов, поэтому ручные
  задержки (`waitForTimeout`) не нужны.

### Паттерн `let ... ; beforeEach`

Во многих файлах видно такое:

```ts
let inventoryPage: InventoryPage;         // объявили переменную заранее

test.beforeEach(async ({ page }) => {
  inventoryPage = new InventoryPage(page); // создали перед каждым тестом
  ...
});

test('...', async () => {
  await inventoryPage.addToCart(...);      // используем в тесте
});
```

`let` объявляет переменную на уровне группы, а в `beforeEach` она пересоздаётся
для каждого теста со свежим `page`. Так объект доступен во всех тестах, но при
этом всегда чистый.

## login.spec.ts — Тикет 1: Login regression

Файл: [login.spec.ts](login.spec.ts). Проверяет, что вход работает только с
валидными данными, а невалидные показывают правильные ошибки.

**`beforeEach`:** создаёт `LoginPage` и открывает страницу логина (`goto`).
Логин здесь **не** делается — каждый тест логинится по-своему.

### Тесты

| Тест | Что проверяет |
|------|---------------|
| `logs in successfully with valid credentials` | `standard_user` входит и попадает на страницу товаров |
| `shows an error with wrong credentials` | неверный пароль → ошибка «Username and password do not match» |
| `shows an error when the form is empty` | пустая форма (сразу клик по кнопке) → ошибка «Username is required» |
| `blocks a locked-out user` | `locked_out_user` → ошибка «this user has been locked out» |

> В тесте с пустой формой вызывается `loginPage.loginButton.click()` напрямую —
> это доступ к локатору без метода `login()`, потому что заполнять поля не нужно.

## cart.spec.ts — Тикет 2: Cart behavior

Файл: [cart.spec.ts](cart.spec.ts). Проверяет, что корзина корректно обновляется
при добавлении и удалении товаров.

**`beforeEach`:** создаёт `LoginPage` и `InventoryPage`, логинится под
`standard_user` и проверяет вход. То есть каждый тест стартует уже на странице
товаров.

### Тесты

| Тест | Что проверяет |
|------|---------------|
| `adds a product to the cart` | после добавления бейдж показывает «1» |
| `removes a product from the cart` | добавили → удалили → бейдж исчез |
| `adds three products and removes one` | 3 товара → бейдж «3», удалили один → «2» |
| `cart contents survive a page reload` | после `page.reload()` товар остаётся в корзине |
| `shows added products on the cart page` | на странице корзины виден нужный товар («Sauce Labs Backpack») |

> В последнем тесте создаётся `CartPage` прямо внутри теста, потому что переход
> на страницу корзины происходит по ходу сценария (`openCart`), а не в
> `beforeEach`.

## checkout.spec.ts — Тикет 3: Checkout flow

Файл: [checkout.spec.ts](checkout.spec.ts). Самый длинный сценарий — полное
оформление заказа. Использует `test.step` для разбивки на шаги.

**`beforeEach`:** логинится, **добавляет рюкзак в корзину** и **открывает
корзину**. То есть каждый тест начинается уже на странице корзины с товаром.

### Тесты

**`completes an order end to end`** — заказ от начала до конца. Разбит на
`test.step`, чтобы в отчёте было видно, на каком именно шаге всё упало:
1. `start checkout from the cart` — жмём Checkout;
2. `enter customer information` — заполняем имя/фамилию/индекс;
3. `overview lists the selected product` — проверяем, что рюкзак есть в обзоре;
4. `finish the order and see the confirmation` — жмём Finish и проверяем
   «Thank you for your order!».

**`requires a first name on the information form`** — негативный кейс: оставляем
имя пустым (`''`) → проверяем ошибку валидации «First Name is required».

> **Что такое `test.step`?** Это способ сгруппировать действия внутри одного
> теста в именованные шаги. На результат теста не влияет, но в отчёте и трейсе
> шаги видны отдельно — удобно для длинных сценариев, чтобы сразу понять, где
> проблема.

## sorting.spec.ts — Тикет 4 (bonus): Product sorting

Файл: [sorting.spec.ts](sorting.spec.ts). Проверяет сортировку товаров по цене
(от дешёвых к дорогим). Самый «алгоритмический» тест.

**`beforeEach`:** логинится под `standard_user` — стартуем на странице товаров.

### Тест

**`sorts products by price low to high`:**
1. выбираем сортировку `lohi` (цена по возрастанию);
2. собираем все цены со страницы в массив чисел (`getPrices`);
3. делаем копию массива и сортируем её вручную по возрастанию;
4. `expect(prices).toEqual(sorted)` — проверяем, что цены **на странице** уже
   идут в том же порядке, что и вручную отсортированный массив.

> Почему сравниваем с копией, а не «просто смотрим что отсортировано»? Playwright
> не умеет проверять «отсортированность» одной командой. Поэтому берём то, что
> реально отдал сайт, и сравниваем с эталоном, который отсортировали сами.
> `[...prices]` — копия, чтобы `.sort()` не менял исходный массив.

## Связь тестов и Page Objects

Тесты не работают с локаторами напрямую (кроме редких случаев вроде
`loginButton.click()`) — они вызывают методы Page Object'ов. Разбор самих
страниц — в [../pages/README.md](../pages/README.md). Тестовые данные (юзеры,
товары, тексты ошибок) — в [../test-data/users.ts](../test-data/users.ts).
